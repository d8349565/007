# 一、拆分总原则

## 1. 先清洗，再拆分

顺序不能反：

```
原文
→ 提取正文
→ 去噪
→ 规范化
→ 拆分
→ 抽取
```

不要直接拿原始 HTML 或 OCR 文本去切。
 否则会把：

- 广告
- 导航
- 免责声明
- 图片说明
- 页脚版权
- 联系方式

一起切进去，纯浪费 token。



# 二、适合你项目的拆分层级

我建议你采用 **四级拆分法**。

## 第 1 级：文档级

判断这篇文章是否需要拆。

规则建议：

- 正文 <= 1200 中文字：**不拆**

- 1200～2500：**优先拆 2 块**

- > 2500：**按段落和小标题拆**

这是第一道成本控制。

------

## 第 2 级：章节级

优先按这些边界切：

- 标题
- 小标题
- 编号标题
- 表格前后说明
- 列表项

比如：

```
一、市场概况
二、竞争格局
三、企业动态
四、风险提示
```

这种天然就是好的 chunk 边界。

原因很简单：
 **同一章节内部，主题通常一致。**

------

## 第 3 级：段落级

如果没有明显章节，就按段落组装。

建议：

- 以自然段为基本单元
- 逐段累加
- 达到阈值再切

而不是先把全文拼成一坨再按字数砍。

------

## 第 4 级：句群级

当段落过长时，再在句号、分号、冒号后切。

优先断点：

- `。`
- `；`
- `：`
- `！`
- `？`

次优断点：

- `，`
   仅在句子极长时用，平时别用。

# 三、推荐的 chunk 设计

## 1. 每块不要太小

太小的问题：

- prompt 重复成本高
- 事实上下文丢失
- 去重更麻烦

建议：

- **最小块：300～500 中文字**
- 小于 200 字的块，通常没必要单独送 LLM

------

## 2. 每块不要太大

太大问题：

- 抽取不稳定
- 容易漏信息
- 输出混乱

建议：

- **理想块：500～900 中文字**
- **上限块：1200～1500 中文字**

对资讯文章来说，这个区间最实用。

------

## 3. 相邻块要保留少量 overlap

这个很重要。
 因为有些事实会跨段：

```
上段给主体
下段给数值
再下段给同比
```

如果完全硬切，容易漏。

建议 overlap：

- **50～120 中文字**
- 或 **1 个自然段**
- 不要超过整块的 20%

overlap 太多会重复抽取，增加去重负担。

------

# 四、适合资讯抽取的拆分策略

## 策略 A：短文直通

适合：

- 快讯
- 简讯
- 公众号短文
- 800～1200 字文章

做法：

```
清洗后整篇送 Extract
```

优点：

- token 最省
- 不需要合并
- 不容易断义

这是默认首选。

------

## 策略 B：按段落组块

适合：

- 1200～3000 字文章
- 普通行业分析
- 财经资讯

做法：

- 以段落为单位累加
- 每块达到 600～900 字切出
- 保留 1 段 overlap

这是你最该用的主策略。

------

## 策略 C：按主题块拆

适合：

- 研报
- 深度长文
- 明显分章节文档
- PDF 报告

做法：

- 先识别标题层级
- 每个小节单独成块
- 过大的小节再做二次段落切分

这比纯字数切更稳。

------

# 五、拆分时必须保留的元数据

每个 chunk 不能只有 text。
 至少要带这些字段：

```
{
  "doc_id": "article_001",
  "chunk_id": "article_001_chunk_03",
  "title": "文章标题",
  "section": "竞争格局",
  "paragraph_start": 8,
  "paragraph_end": 12,
  "char_count": 742,
  "text": "......"
}
```

至少保留：

- `doc_id`
- `chunk_id`
- `title`
- `section`
- `text`
- `position/order`

否则后面没法：

- 回溯来源
- 合并去重
- 定位证据
- 做顺序判断

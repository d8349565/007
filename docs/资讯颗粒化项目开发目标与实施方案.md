# 资讯颗粒化项目开发目标与实施方案

## 一、文档定位

本文档用于定义“资讯颗粒化项目”的第一版落地方案，覆盖以下内容：

- 项目开发目标
- 整体架构设计
- 数据模型设计
- 抽取链路设计
- 审核机制设计
- SQLite / PostgreSQL 表结构设计

说明：

- 所有英文名称、表名、字段名、状态名、关系名保持不变
- 在原有英文术语基础上补充中文注释，便于开发、沟通和后续扩展
- 本方案优先服务于 MVP 阶段，目标是先跑通“文章 → 证据 → 事实原子 → 审核 → 查询分析”

---

## 二、项目开发目标

### 2.1 总体目标

构建一套面向行业资讯、公告、研报、新闻、公众号文章等非结构化文本的数据颗粒化系统，将原始文本中的关键信息抽取为可复用、可追溯、可审核的结构化事实，并逐步沉淀为轻量关系网络。

### 2.2 业务目标

本项目的核心不是做普通摘要，而是做“事实沉淀”。

也就是说，系统最终要支持：

- 按 company、brand、product、policy、event 等对象查询历史事实
- 按时间维度整理 revenue、sales volume、capacity、price change、investment 等指标变化
- 构建企业画像、事件时间轴、竞争格局、价格传导链、政策影响链
- 所有结论都能追溯到原始 evidence，不做黑箱分析

### 2.3 MVP 阶段目标

第一阶段不追求重型知识图谱，不追求复杂多智能体，而是先把以下链路跑通：

`source_document → document_chunk / document_sentence → evidence_span → fact_atom → review → query`

结论：

MVP 的主资产应是 `fact_atom`，即“事实原子表”，而不是图数据库本身。

## 三、整体架构

建议采用六层架构：

### 3.1 `Document Layer`（文档层）

用于存储原始信息源，包括：

- 新闻
- 公告
- 年报
- 研报
- 微信公众号文章
- PDF 文本
- 手工录入内容

核心目标是保留原始内容、来源、标题、时间、URL、哈希值，便于去重与追溯。

### 3.2 `Segmentation Layer`（切分层）

用于把原始文档拆分为：

- `document_chunk`：段落级或块级文本
- `document_sentence`：句子级文本
- `evidence_span`：用于抽取的证据片段

这一层的目标不是做知识存储，而是做“结构化抽取前的最小工作单元”。

### 3.3 `Extraction Layer`（抽取层）

用于从证据片段中抽出结构化事实，主表为：

- `fact_atom`

这是全系统最核心的表。

### 3.4 `Entity Layer`（实体层）

用于管理标准实体及别名，包括：

- `entity`
- `entity_alias`

目标是解决不同写法、简称、中英文名称不统一的问题。

### 3.5 `Relation Layer`（关系层）

用于从已抽取且经过审核的事实中归纳关系边，包括：

- `relation`

这一层是“轻关系图”，不是一开始就上完整知识图谱。

### 3.6 `Review Layer`（审核层）

用于管理抽取任务、审核动作、冲突记录，包括：

- `extraction_task`
- `review_log`
- `conflict_case`

这一层决定数据能否长期可用。

## 四、核心设计原则

### 4.1 `fact_atom` 是主表，不是 `relation`

原因很简单：

你后面真正要做的分析，大多数本质上都落在“事实”上，而不是直接落在“边”上。

例如：

- 某公司在某期间 revenue 是多少
- 某企业新增 capacity 是多少
- 某政策何时发布
- 某产品何时涨价
- 某公司何时并购某对象

这些首先都是事实原子，然后才可能衍生关系。

### 4.2 所有事实必须挂 `evidence`

每一条结构化事实都必须能够回到：

- 哪篇文档
- 哪个 chunk
- 哪一句 sentence
- 哪段 evidence_span

否则后期做分析、核查、纠错会失控。

### 4.3 关系从事实中长出来

不要直接从全文强行抽关系。

更稳的做法是：

1. 先抽 `subject / predicate / object / value / time / location`
2. 再根据这些结构化字段生成 `relation`

这样精度更高，也更方便审核。

### 4.4 先做轻量 schema，再做扩展

MVP 阶段不要一次性设计 100 个 fact_type。

第一版应只覆盖最常见、最有复用价值的事实类型。

## 五、数据模型设计

## 5.1 `source_document`

中文说明：原始文档主表，用于保存资讯来源、标题、正文、时间和去重信息。

建议字段说明：

- `id`：主键
- `source_type`：来源类型，例如 news / annual_report / wechat / research / pdf / manual
- `source_name`：来源名称，例如媒体名、研究机构名、公众号名
- `title`：标题
- `author`：作者
- `url`：原始链接
- `publish_time`：发布时间
- `crawl_time`：抓取时间
- `lang`：语言，默认 `zh`
- `raw_text`：原始正文文本
- `content_hash`：内容哈希，用于去重
- `status`：文档状态，例如 `ACTIVE`

用途：

- 文档追溯
- 重复检测
- 回源核查

---

## 5.2 `document_chunk`

中文说明：文档块表，用于保存段落级或块级切分结果。

建议字段说明：

- `id`：主键
- `document_id`：所属文档 ID
- `chunk_index`：文档中的块序号
- `chunk_text`：块文本
- `char_start`：原文起始字符位置
- `char_end`：原文结束字符位置

用途：

- 分块抽取
- 上下文定位
- 构建证据范围

---

## 5.3 `document_sentence`

中文说明：句子表，用于保存句子级切分结果。

建议字段说明：

- `id`：主键
- `document_id`：所属文档 ID
- `chunk_id`：所属 chunk ID
- `sentence_index`：句序号
- `sentence_text`：句子文本
- `char_start`：原文起始字符位置
- `char_end`：原文结束字符位置

用途：

- 句级 evidence 定位
- 细粒度抽取
- 审核展示

---

## 5.4 `evidence_span`

中文说明：证据片段表，用于保存真正参与抽取的证据范围。

建议字段说明：

- `id`：主键
- `document_id`：所属文档 ID
- `chunk_id`：所属 chunk ID
- `sentence_id_start`：起始句 ID
- `sentence_id_end`：结束句 ID
- `evidence_text`：证据文本
- `char_start`：原文起始字符位置
- `char_end`：原文结束字符位置
- `extraction_task_id`：关联的抽取任务 ID

用途：

- 抽取输入单元
- 审核展示单元
- 证据追溯单元

---

## 5.5 `entity`

中文说明：标准实体主表，用于统一主体、对象、地点、政策等实体。

建议字段说明：

- `id`：主键
- `entity_type`：实体类型
- `canonical_name`：标准名称
- `normalized_name`：标准化名称，用于匹配和去重
- `description`：实体说明
- `country`：国家
- `region`：区域
- `is_active`：是否有效
- `created_at`：创建时间
- `updated_at`：更新时间

建议第一版 `entity_type`：

- `COMPANY`：公司
- `BRAND`：品牌
- `PRODUCT`：产品
- `PERSON`：人物
- `ORG`：组织
- `LOCATION`：地点
- `POLICY`：政策
- `EVENT`：事件
- `METRIC`：指标
- `MATERIAL`：材料
- `PLANT`：工厂
- `VEHICLE_MODEL`：车型

---

## 5.6 `entity_alias`

中文说明：实体别名表，用于保存简称、中英文名、旧名称、股票代码等。

建议字段说明：

- `id`：主键
- `entity_id`：所属标准实体 ID
- `alias_name`：别名
- `alias_type`：别名类型，例如 `abbr / zh / en / old_name / ticker / custom`
- `source`：别名来源

用途：

- 实体标准化
- 模糊匹配
- 中英文映射

---

## 5.7 `fact_atom`

中文说明：事实原子表，是系统最核心的数据表。

建议字段说明：

- `id`：主键
- `document_id`：来源文档 ID
- `evidence_span_id`：证据片段 ID
- `fact_type`：事实类型
- `subject_entity_id`：主体实体 ID
- `predicate`：谓词，表示动作或关系
- `object_entity_id`：对象实体 ID
- `object_text`：对象原始文本，未标准化时保留
- `value_num`：数值型结果
- `value_text`：原始数值文本
- `unit`：单位
- `currency`：币种
- `time_expr`：原始时间表达
- `time_start`：标准化开始时间
- `time_end`：标准化结束时间
- `location_entity_id`：地点实体 ID
- `qualifier_json`：附加限定信息，使用 JSON 文本保存
- `confidence_score`：抽取置信度
- `extraction_model`：使用的抽取模型
- `extraction_version`：抽取版本
- `review_status`：审核状态
- `reviewer`：审核人
- `review_note`：审核备注
- `created_at`：创建时间
- `updated_at`：更新时间

说明：

`fact_atom` 的设计目标是同时兼容以下几类事实：

- 数值事实
- 事件事实
- 指标事实
- 关系事实
- 时间事实

例如：

- 某公司 2025 年 revenue 为 10 亿元
- 某企业在常州新增 5 万吨 capacity
- 某政策于 2026-04-01 发布
- 某公司与某客户签署合作协议

## 六、推荐的 `fact_type` 体系

第一版建议只保留以下 12 类，避免过早复杂化：

- `FINANCIAL_METRIC`：财务指标
- `SALES_VOLUME`：销量
- `CAPACITY`：产能
- `PRICE_CHANGE`：价格变动
- `INVESTMENT`：投资
- `EXPANSION`：扩产 / 扩建
- `NEW_PRODUCT`：新品发布
- `COOPERATION`：合作
- `MNA`：并购
- `POLICY_RELEASE`：政策发布
- `CERTIFICATION`：认证
- `MARKET_SHARE`：市场份额

说明：

每一种 `fact_type` 后续都应有各自的字段抽取模板，不建议一个 Prompt 处理所有类型。

## 七、抽取链路设计

建议采用七步链路。

## 7.1 Step 1：文档入库

输入：

- 网页文章
- PDF
- 公众号文章
- 手工输入内容

处理动作：

- 抽正文
- 去 HTML
- 记录来源和发布时间
- 生成 `content_hash`
- 去重入库

输出：

- `source_document`

---

## 7.2 Step 2：文本切分

处理动作：

- 切 `document_chunk`
- 切 `document_sentence`
- 建立字符位置映射

输出：

- `document_chunk`
- `document_sentence`

说明：

这里的 chunk 是处理单元，不是知识单元。

---

## 7.3 Step 3：候选证据发现

目标：

从句子或多个句子中找出“值得抽取”的片段。

可采用两种方式：

### 方式 A：规则预筛

通过数字、单位、时间、关键词筛出候选句子，例如：

- 同比增长
- 亿元
- 万吨
- 产能
- 签署
- 投产
- 涨价
- 发布
- 拟建

### 方式 B：LLM 句子筛选

判断一句话是否包含可结构化事实。

输出：

- `evidence_span`

---

## 7.4 Step 4：事实原子抽取

目标：

对每个 `evidence_span` 进行结构化抽取，产出标准 JSON，再写入 `fact_atom`。

抽取要求：

- 只抽“证据明确支持”的内容
- 不允许补脑
- 不允许根据常识推断

建议抽取字段：

- `fact_type`
- `subject`
- `predicate`
- `object`
- `value_num`
- `value_text`
- `unit`
- `currency`
- `time_expr`
- `location`
- `qualifiers`
- `confidence`

---

## 7.5 Step 5：实体标准化

目标：

把抽出的 `subject`、`object`、`location` 映射到 `entity`。

建议流程：

1. 精确匹配
2. 别名匹配
3. 标准化匹配
4. 模糊匹配
5. LLM 辅助消歧
6. 未命中则生成候选实体

输出：

- 更新 `entity`
- 更新 `entity_alias`
- 回填 `fact_atom.subject_entity_id / object_entity_id / location_entity_id`

---

## 7.6 Step 6：关系生成

目标：

从高置信度或已审核通过的 `fact_atom` 中生成 `relation`。

说明：

关系生成是二次加工，不应该直接跳过事实层。

---

## 7.7 Step 7：冲突检测与审核

目标：

发现同一主体、同一指标、同一期间下的多版本事实，并交由人工决策。

例如冲突键可以定义为：

`subject + fact_type + predicate + time_expr + qualifier`

输出：

- `conflict_case`
- `review_log`

---

## 八、审核机制设计

结论：

必须采用“自动初筛 + 人工复核”的机制。

纯自动抽取一定会脏。

## 8.1 审核状态建议

建议在 `review_status` 中保留以下状态：

- `PENDING`：待审核
- `AUTO_PASS`：自动通过
- `HUMAN_PASS`：人工通过
- `REJECTED`：驳回
- `UNCERTAIN`：不确定
- `CONFLICT`：冲突

---

## 8.2 自动通过条件

满足以下条件，可考虑自动进入 `AUTO_PASS`：

- 结构完整
- 主体明确
- 数值明确
- 单位明确
- 时间明确
- evidence 支撑充分
- `confidence_score >= 0.90`
- 与现有数据不冲突

---

## 8.3 必须人工审核的情况

以下情况不建议自动放行：

- 金额类事实
- market share 类事实
- yoy / qoq 类事实
- 推断性语言
- 一句多事实
- 主体歧义
- 时间歧义
- 多来源冲突

---

## 8.4 审核动作建议

审核员应至少支持以下动作：

- 通过
- 驳回
- 修改后通过
- 合并到已有 fact
- 标记冲突
- 标记待补证据

---

## 九、SQLite / PostgreSQL 表结构

下面给出可直接落地的一版表结构。英文名称保持不变，适合 SQLite 和 PostgreSQL 做第一版原型。

### 9.1 `source_document`

```sql
CREATE TABLE source_document (
    id TEXT PRIMARY KEY,
    source_type TEXT NOT NULL,
    source_name TEXT,
    title TEXT NOT NULL,
    author TEXT,
    url TEXT,
    publish_time TIMESTAMP,
    crawl_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    lang TEXT DEFAULT 'zh',
    raw_text TEXT NOT NULL,
    content_hash TEXT NOT NULL,
    status TEXT DEFAULT 'ACTIVE'
);

CREATE UNIQUE INDEX idx_source_document_hash
ON source_document(content_hash);

CREATE INDEX idx_source_document_publish_time
ON source_document(publish_time);
```

中文说明：

- 用于保存原始文档及元信息
- `content_hash` 用于去重
- `publish_time` 用于时间轴分析

---

### 9.2 `document_chunk` / `document_sentence` / `evidence_span`

```sql
CREATE TABLE document_chunk (
    id TEXT PRIMARY KEY,
    document_id TEXT NOT NULL,
    chunk_index INTEGER NOT NULL,
    chunk_text TEXT NOT NULL,
    char_start INTEGER,
    char_end INTEGER,
    FOREIGN KEY (document_id) REFERENCES source_document(id)
);

CREATE INDEX idx_document_chunk_doc
ON document_chunk(document_id, chunk_index);

CREATE TABLE document_sentence (
    id TEXT PRIMARY KEY,
    document_id TEXT NOT NULL,
    chunk_id TEXT,
    sentence_index INTEGER NOT NULL,
    sentence_text TEXT NOT NULL,
    char_start INTEGER,
    char_end INTEGER,
    FOREIGN KEY (document_id) REFERENCES source_document(id),
    FOREIGN KEY (chunk_id) REFERENCES document_chunk(id)
);

CREATE INDEX idx_document_sentence_doc
ON document_sentence(document_id, sentence_index);

CREATE TABLE evidence_span (
    id TEXT PRIMARY KEY,
    document_id TEXT NOT NULL,
    chunk_id TEXT,
    sentence_id_start TEXT,
    sentence_id_end TEXT,
    evidence_text TEXT NOT NULL,
    char_start INTEGER,
    char_end INTEGER,
    extraction_task_id TEXT,
    FOREIGN KEY (document_id) REFERENCES source_document(id),
    FOREIGN KEY (chunk_id) REFERENCES document_chunk(id)
);

CREATE INDEX idx_evidence_span_doc
ON evidence_span(document_id);
```

中文说明：

- `document_chunk`：保存块级文本
- `document_sentence`：保存句级文本
- `evidence_span`：保存参与抽取的证据片段

---

### 9.3 `entity` / `entity_alias`

```sql
CREATE TABLE entity (
    id TEXT PRIMARY KEY,
    entity_type TEXT NOT NULL,
    canonical_name TEXT NOT NULL,
```

## 十、推荐的三 Agent 思路

为了提高抽取稳定性，不建议使用一个万能 Prompt。

建议拆成三个 Agent。

### 10.1 `Evidence Finder`

中文说明：证据发现 Agent。

职责：

- 判断一句话或一段话是否包含可结构化事实
- 给出候选 `fact_type`
- 提取最小 `evidence_span`

### 10.2 `Fact Extractor`

中文说明：事实抽取 Agent。

职责：

- 从 `evidence_span` 中提取标准结构字段
- 输出标准 JSON
- 不允许推断

### 10.3 `Reviewer / Validator`

中文说明：审核校验 Agent。

职责：

- 判断抽取内容是否被 evidence 明确支持
- 检查 subject / value / unit / time 是否一致
- 输出 `PASS / REJECT / UNCERTAIN`

这一步对降低脏数据非常关键。

### 第一阶段必须完成

### 第三阶段再做展示分析

- 事件链
- 指标聚合
- 竞争格局分析

结论：

不要一开始就做复杂图谱前端。先把数据质量做起来。

## 十二、最终建议

当前最合适的落地路线是：

**先做“事实原子库 + 证据链 + 审核闭环”，再做“轻关系图 + 分析视图”。**

这是最稳的路径，原因有三点：

1. 数据质量优先，避免后期返工
2. 查询与分析直接依赖 `fact_atom`，业务价值释放更快
3. 后续无论接 SQLite、PostgreSQL、Neo4j 还是其他图层，都有清晰底座
